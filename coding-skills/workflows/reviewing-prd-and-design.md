# PRD 和详细设计评审

## 概述

**评审必须由独立的 Agent 执行，不能由编写者自评。**

评审 Agent 扮演"质疑者"角色，以批判性眼光审查文档，确保：
- 需求清晰、完整、可验证
- 设计可行、风险可控
- 没有遗漏关键细节

## 核心原则

1. **独立评审** — 评审者不能是编写者
2. **批判性思维** — 假设文档有问题，主动寻找
3. **有理有据** — 每个问题都要有具体的修改建议
4. **不放过细节** — 边界条件、异常处理、扩展性都要审查

## 评审流程

```
提交评审 → 评审Agent审查 → 发现问题 → 返回修改
                                         ↑
                          反复修改直到评审通过
                                         ↓
                              评审通过 → 提交用户审批
```

## PRD 评审清单

### 1. 需求完整性

- [ ] 每个功能都有明确的用户故事？
- [ ] 每个功能都有具体的验收标准（Given-When-Then）？
- [ ] 功能优先级（P0/P1/P2）是否合理？
- [ ] 是否有遗漏的边界情况？

### 2. 业务价值

- [ ] 功能解决的是真实用户痛点？
- [ ] 价值主张是否清晰、可衡量？
- [ ] 与现有功能的边界是否清晰？

### 3. 可行性

- [ ] 验收标准是否可以转化为测试用例？
- [ ] 是否有依赖外部系统的地方？
- [ ] 时间线和资源是否合理？

### 4. 非功能需求

- [ ] 性能要求是否明确（响应时间、并发）？
- [ ] 安全要求是否覆盖（认证、授权、数据保护）？
- [ ] 可用性要求是否合理？

## 设计评审清单

### 1. 架构设计

- [ ] 架构是否清晰、易于理解？
- [ ] 是否能支撑当前需求？
- [ ] 是否有扩展性考虑？
- [ ] 技术选型是否有充分理由？

### 2. 模块设计

- [ ] 模块职责是否单一、清晰？
- [ ] 模块间接口是否定义明确？
- [ ] 模块间依赖是否合理？
- [ ] 数据流是否清晰？

### 3. 详细设计

- [ ] API 设计是否 RESTful？
- [ ] 数据库设计是否合理？
- [ ] 错误处理是否完整？
- [ ] 日志和监控是否考虑？

### 4. 风险评估

- [ ] 是否有技术风险？
- [ ] 是否有备选方案？
- [ ] 风险影响和概率评估？
- [ ] 缓解措施是否可行？

### 5. 测试策略

- [ ] 单元测试覆盖率目标是否合理（100%）？
- [ ] 集成测试场景是否覆盖关键路径？
- [ ] 是否有性能测试计划？

## 评审输出格式

```markdown
## 评审报告

**文档：** [PRD/设计文档名称]
**版本：** v0.x.x
**评审时间：** YYYY-MM-DD
**评审结论：** 通过 / 需修改 / 不可行

---

### 发现的问题

#### 🔴 严重问题

1. **[问题描述]**
   - 位置：章节X.X
   - 影响：[对开发/用户的影响]
   - 建议：[具体的修改建议]

#### 🟡 一般问题

2. **[问题描述]**
   - 位置：章节X.X
   - 建议：[修改建议]

#### 💡 优化建议

3. **[建议描述]**
   - 理由：[为什么建议修改]
   - 建议：[可选的改进方案]

---

### 评审结论

| 项目 | 结论 | 说明 |
|------|------|------|
| 完整性 | ✅/⚠️/❌ | [说明] |
| 清晰度 | ✅/⚠️/❌ | [说明] |
| 可行性 | ✅/⚠️/❌ | [说明] |
| 风险控制 | ✅/⚠️/❌ | [说明] |

---

### 下一步

- [ ] 根据以上反馈修改文档
- [ ] 修改后重新提交评审
- [ ] 评审通过后提交用户审批
```

## 评审结论处理

| 结论 | 操作 |
|------|------|
| **通过** | 文档提交用户审批 |
| **需修改** | 将评审报告反馈给编写者，修改后重新评审 |
| **不可行** | 终止项目，重新评估需求或技术方案 |

## 评审通过条件

所有检查项都通过或仅有"优化建议"级别的问题：
- 0 个 🔴 严重问题
- 0 个 🟡 一般问题
- 可选的 💡 优化建议

## 与编写者的协作

1. **评审者发现问题** → 填写评审报告
2. **评审者提交反馈** → 通知编写者
3. **编写者修改文档** → 提交新一轮评审
4. **循环直到通过** → 评审通过后提交审批

## 关键原则

- **评审是建设性的** — 目标是提升文档质量，不是批评
- **问题要有建议** — 每个问题都要有具体的修改方向
- **反复修改是正常的** — 好文档是迭代出来的
- **评审通过后才审批** — 确保文档质量后再提交用户
