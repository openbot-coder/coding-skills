# 编写 PRD 和详细设计

## 适用范围

| 项目规模 | 功能点 | 文档要求 |
|----------|--------|----------|
| **脚本类** | ≤ 50 FP | 无需文档，使用 `script-development` |
| **中小型** | < 1000 FP | 1 份 PRD + 1 份设计文档 |
| **大型** | ≥ 1000 FP | 按模块拆分，每个模块独立 PRD + 设计 |

> **大型项目提示**：功能点 ≥ 1000 时，将系统拆分为多个模块，每个模块独立执行完整的 PRD → 设计 → 实施流程。

## 概述

在头脑风暴获得批准后，将需求转化为 PRD 文档和详细设计文档。

- **PRD** — 回答"做什么"（功能、用户价值、验收标准）
- **详细设计** — 回答"怎么做"（架构、技术方案、组件设计）

## 流程

```
brainstorming → 编写PRD → 评审PRD → 编写设计 → 评审设计 → writing-plans → 执行
```

**重要原则：编写与评审必须由不同的 Agent 执行。**

## 子 Agent 职责

| Agent | 职责 |
|-------|------|
| `writing-prd-and-design` | 编写 PRD 和详细设计文档 |
| `reviewing-prd-and-design` | **独立评审** PRD 和详细设计文档 |

## 评审迭代流程（关键）

评审是一个**编写Agent与Reviewer Agent之间反复修改文档直到通过**的过程：

```
┌─────────────────────────────────────────────────────────────┐
│ 1. writing-prd-and-design Agent 编写文档草稿               │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. reviewing-prd-and-design Agent 评审                      │
└─────────────────────────────────────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
              ▼                               ▼
        ⚠️ 需修改                        ✅ 通过
              │                               │
              ▼                               │
    ┌─────────────────┐                      │
    │ 3. 返回编写Agent │                      │
    │    修改文档      │                      │
    └─────────────────┘                      │
              │                               │
              └───────────────┬───────────────┘
                              ▼
              ┌─────────────────────────────┐
              │ 4. 重新提交给 Reviewer Agent │
              │    继续评审                  │
              └─────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
              ▼                               ▼
        ⚠️ 需修改                        ✅ 通过
              │                               │
              └───────────────┘               │
              （继续修改-评审循环）           │
                                              ▼
                                    进入下一阶段
```

**注意：** 整个迭代过程由 `writing-prd-and-design` Agent 和 `reviewing-prd-and-design` Agent 之间完成，无需用户介入，直到两份文档都评审通过后，才请求用户批准。

## 评审规则

1. **编写者不能评审自己的文档** — 必须由不同的 Agent 执行
2. **评审发现问题时，由编写Agent修改文档**
3. **修改完成后重新提交给Reviewer Agent继续评审**
4. **反复修改-评审循环，直到Reviewer Agent通过**
5. **两份文档都评审通过后，由Reviewer Agent请求用户批准**

## 检查清单

### PRD 阶段

- [ ] `writing-prd-and-design` Agent 编写 PRD 文档
- [ ] `reviewing-prd-and-design` Agent 评审
- [ ] ⚠️ 发现问题 → 返回编写Agent修改 → 重新提交评审
- [ ] ... 反复修改-评审循环直到通过
- [ ] ✅ PRD 评审通过，更新状态为"待审批"

### 设计阶段

- [ ] `writing-prd-and-design` Agent 编写详细设计文档
- [ ] `reviewing-prd-and-design` Agent 评审
- [ ] ⚠️ 发现问题 → 返回编写Agent修改 → 重新提交评审
- [ ] ... 反复修改-评审循环直到通过
- [ ] ✅ 详细设计评审通过，更新状态为"待审批"

### 收尾

- [ ] `reviewing-prd-and-design` Agent 请求用户批准两份文档
- [ ] 用户批准后更新状态为"已批准"
- [ ] 过渡到实施计划 — 调用 `writing-plans` 技能

## 关键原则

- **PRD 关注业务价值** — 回答"为什么"和"做什么"
- **设计关注技术实现** — 回答"怎么做"
- **编写与评审分离** — 由不同的 Agent 执行，编写者不能评审自己的文档
- **评审是必需的** — 文档需要评审通过后才能进入下一阶段
- **文档先行** — 在触碰代码之前完成文档
- **反复修改是正常的** — 好文档是评审-修改迭代出来的
