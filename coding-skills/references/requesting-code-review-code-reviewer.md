## 审查清单

**代码质量：**
- 关注点是否清晰分离？
- 错误处理是否得当？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 边界情况是否处理？

**架构：**
- 设计决策是否合理？
- 是否考虑了可扩展性？
- 性能影响如何？
- 是否存在安全隐患？

**测试：**
- 测试是否真正验证了逻辑（而非 mock）？
- 边界情况是否覆盖？
- 需要集成测试的地方是否有？
- 所有测试是否通过？

**需求：**
- 计划中的所有需求是否满足？
- 实现是否符合规格？
- 是否没有范围蔓延？
- 破坏性变更是否已记录？

**生产就绪性：**
- 迁移策略（如有 schema 变更）？
- 是否考虑了向后兼容性？
- 文档是否完整？
- 是否没有明显的 bug？

## 关键规则

**要做：**
- 按实际严重程度分类（不要把所有问题都标为严重）
- 要具体（文件:行号，不要含糊）
- 解释问题为什么重要
- 肯定优点
- 给出明确结论

**不要做：**
- 不检查就说"看起来不错"
- 把吹毛求疵的问题标为严重
- 对没有审查的代码给出反馈
- 含糊其辞（"改进错误处理"）
- 回避给出明确结论
