# 调试与验证

## 概述

阶段4（verify）是验证阶段，用于确认阶段3（execute）中完成的任务是否正确实现。

**两个核心能力：**
1. **验证**：确认功能按预期工作
2. **调试**：遇到问题时系统化地找到根因并修复

**核心原则：** 验证通过后才能进入下一阶段。遇到问题必须先找到根因再修复。

## 何时使用

- 阶段3完成后，进入阶段4验证
- 验证过程中发现任何问题
- 测试失败或意外行为
- 需要确保质量标准达标

# 第一部分：验证（verification-before-completion）

## 铁律

```
在进入下一阶段之前，必须验证已完成的工作
```

跳过验证 = 承诺交付缺陷。

## 验证清单

对于每个已完成的子任务/功能点，验证以下内容：

### 1. 功能正确性

| 检查项 | 标准 | 状态 |
|--------|------|------|
| 核心功能 | 功能按提案描述工作 | ☐ |
| 用户交互 | 界面响应正确 | ☐ |
| 数据处理 | 数据正确保存/读取 | ☐ |
| 错误处理 | 异常情况正确处理 | ☐ |

### 2. 测试质量

| 检查项 | 标准 | 状态 |
|--------|------|------|
| 单元测试 | 所有功能点有测试覆盖 | ☐ |
| 测试通过 | pytest 全部通过 | ☐ |
| 覆盖率 | 单元测试覆盖率 ≥ 90% | ☐ |
| 未覆盖行 | 已标注 # UNCOVERED 原因 | ☐ |

### 3. 代码质量

| 检查项 | 标准 | 状态 |
|--------|------|------|
| 代码风格 | 符合项目规范 | ☐ |
| 无警告 | 编译器/静态分析无警告 | ☐ |
| 文档 | 公共 API 有文档 | ☐ |

### 4. 提交状态

| 检查项 | 标准 | 状态 |
|--------|------|------|
| Git 状态 | 代码已提交 | ☐ |
| 提交信息 | 清晰描述变更内容 | ☐ |

## 验证流程

1. **逐个任务验证**：按任务清单顺序验证每个任务
2. **记录结果**：在验证结果表中记录每个检查项的状态
3. **发现问题时**：进入调试流程
4. **全部通过后**：进入阶段5（archive）

# 第二部分：调试（systematic-debugging）

## 铁律

```
没有根因调查就不能修复
```

随意修复浪费时间并制造新 bug。快速补丁掩盖潜在问题。

**核心原则：** 总是先找到根因再尝试修复。只修症状是失败。

## 何时使用

- 验证过程中发现任何问题
- 测试失败
- 功能不按预期工作
- 遇到 bug

**特别在以下情况使用：**
- 问题看起来简单时（简单 bug 也有根因）
- "只需一个快速修复"似乎很明显时
- 你已经尝试了多种修复时
- 上一次修复不起作用时

## 四个阶段

你必须完成每个阶段才能进入下一个。

### 第 1 阶段：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **一致地复现**
   - 你能可靠地触发它吗？
   - 确切步骤是什么？
   - 每次都发生吗？
   - 如果不可复现 → 收集更多数据，不要猜测

3. **检查最近的更改**
   - 什么改变可能导致了这个？
   - Git diff、最近提交

4. **在多组件系统中收集证据**

   对于每个组件边界：
   - 记录什么数据进入组件
   - 记录什么数据离开组件
   - 验证环境/配置传播
   - 检查每层状态

### 第 2 阶段：模式分析

**在修复之前找到模式：**

1. **找到工作示例**
   - 在同一代码库中找到类似的正常工作的代码

2. **对照参考比较**
   - 如果有参考实现，完整阅读

3. **识别差异**
   - 工作的和损坏的之间有什么不同？

4. **理解依赖**
   - 这需要什么其他组件？

### 第 3 阶段：假设与测试

**科学方法：**

1. **形成单一假设**
   - 清楚陈述："我认为 X 是根因，因为 Y"
   - 写下来

2. **最小化测试**
   - 做最小的可能更改来测试假设
   - 一次一个变量

3. **在继续之前验证**
   - 有效吗？是 → 第 4 阶段
   - 没效果？形成新假设

### 第 4 阶段：实施

**修复根因，而非症状：**

1. **创建失败的测试用例**
   - 最简单的可能复现
   - 修复前必须有

2. **实施单一修复**
   - 解决识别出的根因
   - 一次一个更改

3. **验证修复**
   - 测试现在通过了吗？
   - 没有其他测试被破坏？

4. **如果修复不起作用**
   - 停止
   - 计数：你尝试了多少次修复？
   - 如果 < 3 次：返回第 1 阶段
   - **如果 ≥ 3 次：停下来质疑架构**

## 红线 - 停下并遵循流程

如果你发现自己在想：
- "先快速修复，稍后调查"
- "就试试改 X 看看是否有效"
- "跳过测试，我手动验证"
- "可能是 X，让我修复那个"
- **"再试一次修复"（当已经尝试 2 次以上时）**

**所有这些意味着：停下。返回第 1 阶段。**

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 阅读错误、复现、检查更改 | 理解什么和为什么 |
| **2. 模式** | 找到工作示例、比较 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或新假设 |
| **4. 实施** | 创建测试、修复、验证 | Bug 解决，测试通过 |

# 第三部分：整合流程

## 阶段4完整流程

```
开始验证
    ↓
遍历每个任务
    ↓
验证清单检查
    ↓
发现问题的？
    ├── 是 → 进入调试流程（四个阶段）
    │         ↓
    │      修复后重新验证
    │         ↓
    │      继续下一个任务
    │
    └── 否 → 继续下一个任务
    ↓
所有任务通过
    ↓
进入阶段5（archive）
```

|------|---------|---------|---------|---------|
| - | - | - | - | - |

# 第四部分：与 TDD 的关系

调试与 TDD（test-driven-development）紧密配合：

| TDD 阶段 | 调试对应 |
|---------|---------|
| 红（写失败测试） | 验证发现测试失败 → 调试流程开始 |
| 绿（写最少代码） | 第4阶段：实施修复 |
| 重构 | 修复后重新验证 |

**关键点：** 即使在 TDD 循环中，也要遵循"先根因后修复"的原则。
