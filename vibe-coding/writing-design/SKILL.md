# 需求调研与访谈

## 概述

通过自然的协作对话，帮助将想法转化为完整的设计内容。

首先了解当前项目上下文，然后逐一提问来细化想法。一旦理解了需求，展示解决方案并获得用户批准。

> **参考文档**：
> - [架构设计原则](../references/ARCHITECTURE-PRINCIPLES.md) — 术语表、核心原则、设计检查点
> - [接口设计方法](../references/ARCHITECTURE-INTERFACE-DESIGN.md) — Design It Twice
> - [调研记录格式](../references/SURVEY-RECORDS.md) — 调研记录模板

<HARD-GATE>
在展示解决方案并获得用户批准之前，不得调用任何实现技能、编写任何代码、搭建任何项目或采取任何实现行动。
</HARD-GATE>

## 调研记录

**所有调研对话必须实时记录到 `{name}-survey-records.md`。**

详见 [调研记录格式](../references/SURVEY-RECORDS.md)：
- 保留原始对话，不改写
- 记录术语澄清、场景测试、决策、待确认事项
- 每次调研追加新章节，记录日期和访谈人

## 检查清单

1. **探索项目上下文** — 检查文件、文档、docs/vibe-coding/CONTEXT.md
2. **调研对话** — 一次问一个问题，实时记录到 survey-records
3. **挑战模糊术语** — 发现模糊词立即澄清，更新记录
4. **场景测试** — 用具体场景验证理解
5. **提出 2-3 种方案** — 包含权衡和推荐
6. **展示解决方案** — 按复杂度分节，每节后获得用户批准
7. **架构设计** — 模块划分、接口设计、深度评估
8. **过渡到审查** — 完成 writing-design，准备进入 review-design

## 流程

### Step 1: 探索项目上下文

1. 检查当前项目状态：
   - 查看 `docs/vibe-coding/` 下的现有设计
   - 查找 `docs/vibe-coding/CONTEXT.md`（领域术语定义）
   - 查找 `docs/vibe-coding/[changes-name]-survey-records.md`（调研记录）
2. 了解现有术语和决策，避免重复讨论

### Step 2: 调研对话

**原则：一次问一个问题，等待回答后再继续。**

#### 提问维度

1. **目的**：想解决什么问题？
2. **约束**：有什么限制条件？
3. **成功标准**：怎样才算"完成了"？
4. **范围**：包含什么？不包含什么？
5. **用户**：谁会使用这个功能？
6. **场景**：典型使用场景是什么？

#### 挑战模糊术语

当用户使用模糊或歧义词汇时，立即挑战：

> "你刚才说'用户'——这里是指下单的用户还是商家？"

使用 [CONTEXT-FORMAT](./CONTEXT-FORMAT.md) 格式记录澄清后的术语：
- 原术语：用户模糊使用的词
- 澄清后：达成共识的精确术语
- 说明：为什么区分

#### 场景测试

用具体场景验证理解：

> "那如果用户在下单后还没付款就取消了，通知怎么发？"

将场景和结论记录到 survey-records。

### Step 3: 提出方案

- 提出 2-3 种不同的方案
- 分析每种方案的权衡
- 给出推荐和理由
- 以对话方式展示，逐一确认

### Step 4: 展示解决方案

- 按复杂度分节展示（简单几句话，复杂 200-300 字）
- 每节后问是否正确
- 涵盖：架构、组件、数据流、错误处理、测试

### Step 5: 架构设计

> 详见 [架构设计原则](../references/ARCHITECTURE-PRINCIPLES.md)

设计时关注：
- **模块划分**：职责清晰，无透传模块
- **接口设计**：足够小（3-5 方法），调用者容易理解
- **深度评估**：删除测试（集中复杂性 = 好）
- **接缝位置**：值得抽象才做（至少两个适配器）

### Step 6: 记录调研

调研过程中，实时更新 `{name}-survey-records.md`：

```markdown
## 第N次调研

**日期：** YYYY-MM-DD HH:mm
**访谈人：** {访谈人}
**主题：** {本次调研的主题}

### 对话记录

> **助手：** [问题]
>
> **用户：** [回答]

### 新增术语
| 原术语 | 澄清后 | 说明 |
|--------|--------|------|

### 新增场景
| 场景 | 结论 |
|------|------|

### 新增决策
| 决策 | 理由 |
|------|------|

### 待确认
| 问题 | 状态 |
|------|------|
```

## 关键原则

- **一次一个问题** — 不要用多个问题淹没用户
- **优先选择题** — 可能的话比开放式更容易回答
- **挑战模糊术语** — 发现歧义立即澄清，不累积
- **场景测试** — 用具体场景验证理解，不是抽象讨论
- **代码交叉验证** — 用户说的和代码不一致时指出
- **实时记录** — 对话同时记录，不事后补写
- **严格执行 YAGNI** — 移除不必要的功能
- **探索替代方案** — 提出 2-3 种方案

## 术语澄清处理

### 发现模糊术语

1. **立即指出**：
   > "你说的'account'是指买家账户还是支付账户？"

2. **澄清后记录**：
   更新 survey-records 的术语澄清记录

3. **更新 `docs/vibe-coding/CONTEXT.md`**（如存在）：
   ```markdown
   **Account**:
   A payment instrument used to settle transactions.
   _Avoid_: wallet, balance
   ```

### 创建 CONTEXT.md

如果项目没有 `docs/vibe-coding/CONTEXT.md`，在第一个术语澄清时创建：

```markdown
# {项目名} 领域模型

## 语言

**{术语}**:
{一句话定义}
_Avoid_: {避免使用的同义词}

## 关系

- **{术语A}** 产生 **{术语B}**

## 示例对话

> **Dev:** "{问题}"
> **领域专家:** "{回答}"
```

## 与 vibe-coding 的衔接

```
writing-design → review-design → 任务拆解 → 代码执行 → 测试验证 → 需求归档
```

**操作：**
1. 运行 `python scripts/design.py --name <变更名称>` 创建设计目录
2. 创建 `{name}-survey-records.md` 开始记录调研
3. 将需求要点和设计方案填入 `{name}-design.md`
4. 完成调研和设计后，进入 review-design 阶段

## 输出文件

| 文件 | 说明 |
|------|------|
| `docs/vibe-coding/{name}-design.md` | 设计文档（目标、背景、成功标准、架构设计） |
| `docs/vibe-coding/{name}-survey-records.md` | 调研记录（原始对话、术语澄清、场景测试、决策） |
| `docs/vibe-coding/CONTEXT.md` | 领域术语（首次澄清术语时创建/更新） |
