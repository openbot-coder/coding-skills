# 反模式

## 开发反模式

- ❌ 不经提案直接编码
- ❌ 没有计划就实施
- ❌ 跳过验证声称"完成"
- ❌ "这太简单了不需要提案"
- ❌ 一次处理多个任务

## 借口反驳表

| 借口 | 反驳 |
|------|------|
| "这太简单了不需要提案" | 简单项目正是假设导致浪费的地方。提案可以简短，但必须写下来。 |
| "我先写代码再说" | 没有目标的代码是负债。先定义成功标准，再动手。 |
| "这只是小改动" | 最大的 bug 通常来自"小改动"。走一遍流程很快。 |
| "运行没问题，我测试过了" | 你测试的是 happy path。用户会走边界条件和错误路径。 |
| "以后再验证" | 验证的成本随时间指数增长。现在验证，5 分钟；以后验证，5 小时。 |

## 警告信号

如果发现自己有以下行为，立即停止并纠正：

| 警告信号 | 处理方式 |
|----------|----------|
| "顺便把这个也修了吧" | 记录到 TODO，不在本任务处理 |
| "这个函数有问题，我先改改" | 确认是否影响当前任务，影响则记录，不影响则跳过 |
| "发现一个新需求" | 暂停，记录到 backlog，回到当前任务 |
| "这个架构不太好，重构一下" | 不重构，除非明确被要求 |

## 规则

1. **任何开发任务始终先通过此技能路由。**
2. **绝不跳过阶段。** 即使任务"看起来很简单"。
3. **阶段可以回退。** 执行中发现问题可以回到计划阶段。
4. **验证是强制性的。** 没有验证证据就不能声称完成。
5. **宣布阶段：** "阶段 \[N]：\[名称]"
