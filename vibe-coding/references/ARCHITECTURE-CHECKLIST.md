# 架构审查清单

设计审查中的架构质量检查项。发现浅层模块，提出深化机会。

## 审查维度

### 1. 模块设计

- [ ] **模块划分合理吗？**
  - 理解这个功能需要看几个模块？（越少越好）
  - 有没有过度拆分（5+ 个小模块做简单的事）？
  - 有没有不足拆分（一个模块做太多事）？

- [ ] **模块命名是否使用领域术语？**
  - 用 "OrderIntake" 而非 "OrderHandler"
  - 用 "PaymentGateway" 而非 "PayStuff"

- [ ] **有没有浅层模块？**
  - 是否有"只是透传"的模块？
  - 接口是否几乎和实现一样复杂？
  - 删除它会影响调用者吗？（不会 = 浅层）

### 2. 接口设计

- [ ] **接口是否清晰、可测试？**
  - 调用者能通过接口验证行为吗？
  - 接口是否暴露了不必要的内部状态？
  - 测试能否只通过接口验证行为？

- [ ] **接口是否足够小？**
  - 每个模块 3-5 个方法以内？
  - 调用者需要记住多少才能正确使用？

- [ ] **接口约束是否明确？**
  - 前后置条件是否记录？
  - 错误模式是否清晰？
  - 调用顺序有要求吗？

### 3. 深度评估

- [ ] **删除测试**
  - 删除这个模块，复杂性消失还是重现？
  - 模块是否在"挣自己的价值"？
  - 删除后调用者会崩溃吗？

- [ ] **杠杆分布**
  - 哪些模块是"深"的（大量行为 + 小接口）？
  - 哪些模块是"浅"的（几乎没有杠杆）？

- [ ] **测试覆盖潜力**
  - 通过接口能测试多少行为？
  - 是否需要 mock 很多东西才能测试？

### 4. 接缝设计

- [ ] **可扩展点是否预留接缝？**
  - 哪里可能需要替换/扩展？
  - 是否真的需要两个以上适配器？

- [ ] **接缝位置是否合理？**
  - 是否在变化频率高的地方？
  - 深度模块的内部不要随便拆

- [ ] **内部 vs 外部接缝**
  - 内部接缝是否暴露到外部了？（不应该）
  - 内部实现是否通过接缝隐藏好了？

### 5. 一致性检查

- [ ] **与现有架构风格一致？**
  - 新模块是否遵循现有模式？
  - 命名是否与项目术语一致？

- [ ] **与现有 ADR/决策冲突？**
  - 是否有现有决策禁止的设计？
  - 是否需要新增 ADR 或更新现有 ADR？

## 审查问题清单

### 基础问题

1. **目标清晰吗？** — 能否用一句话复述？
2. **背景充分吗？** — 为什么现在要做？
3. **成功标准可验证吗？** — 能通过测试验证？
4. **范围明确吗？** — 包含和不包含清晰？
5. **假设合理吗？** — 基于事实而非推测？
6. **风险可控吗？** — 有缓解措施？
7. **有安全风险吗？** — 有无新攻击面？
8. **影响范围已确认吗？** — 用户是否知情同意？

### 架构问题

9. **模块划分合理吗？**
   - 理解这个功能需要看几个模块？
   - 有没有"只是透传"的模块？
   - 模块间是否有不必要的耦合？

10. **接口设计够小吗？**
    - 每个模块的接口有几个方法？
    - 调用者需要记住多少才能正确使用？
    - 有没有把实现细节泄露到接口上？

11. **深度是否平衡？**
    - 删除这个模块会集中复杂性还是分散？
    - 哪些模块是浅的，需要考虑合并？
    - 哪些模块是深的，值得保持？

12. **接缝位置是否合理？**
    - 哪里可能变化需要可替换？
    - 这个接缝是否值得抽象（至少两个适配器）？
    - 内部细节是否隐藏好了？

## 深化候选识别

识别以下问题的模块作为深化候选：

| 问题 | 症状 | 建议 |
|------|------|------|
| 浅层模块 | 接口≈实现，无杠杆 | 合并到调用者或深化 |
| 透传模块 | 删除后调用者无感 | 删除或合并 |
| 过度抽象 | 一个接口只有一个实现 | 删除抽象 |
| 不足抽象 | 一个模块做太多事 | 拆分职责 |
| 耦合泄漏 | 调用者需要知道内部细节 | 隐藏实现 |
