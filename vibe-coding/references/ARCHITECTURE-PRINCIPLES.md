# 架构设计原则

发现架构摩擦、提出深化机会的核心方法论。目标：可测试性和 AI 可导航性。

## 术语表

使用一致术语是架构沟通的基础。不要用"component"、"service"、"API"、"boundary"等模糊词汇。

| 术语 | 定义 | 示例 | 避免用 |
|------|------|------|--------|
| **模块 (Module)** | 有接口和实现的一切（函数、类、包、切片） | `OrderIntake`, `UserRepository` | unit, component, service |
| **接口 (Interface)** | 调用者必须知道的一切（类型、约束、错误模式、顺序） | `save()`, `findById()` | API, signature |
| **实现 (Implementation)** | 模块内部的代码 | - | - |
| **深度 (Depth)** | 大量行为隐藏在简单接口之后 = 高杠杆 | 一个 `Repository` 隐藏所有存储细节 | - |
| **浅层 (Shallow)** | 接口几乎和实现一样复杂 = 低杠杆 | 透传函数、无实际逻辑的包装 | - |
| **接缝 (Seam)** | 可以改变行为而不编辑原地的地方 | 适配器注入点 | boundary |
| **适配器 (Adapter)** | 满足接口的具体实现 | `InMemoryUserRepo`, `PostgresUserRepo` | - |
| **杠杆 (Leverage)** | 调用者从深度获得的能力 | 更多能力/更少学习成本 | - |
| **局部性 (Locality)** | 维护者从深度获得的优势 | 变更集中在一处 | - |

## 核心原则

### 1. 删除测试

> 想象删除这个模块。复杂性消失 = 透传。复杂性在 N 个调用者中重现 = 值得保留。

- **好模块**：删除它，调用者全部崩溃，复杂性集中在一处
- **坏模块**：删除它，调用者毫无感觉，只是把复杂性分散到各处

### 2. 接口即契约

接口包括调用者必须知道的一切：
- 类型签名
- 不变量（调用前/后必须满足的条件）
- 错误模式（可能抛出的异常）
- 顺序约束（调用顺序要求）
- 性能特性（O(n) 还是 O(1)）

### 3. 深度优先

```
深模块：大量行为 + 小接口 = 高杠杆
浅模块：接口 ≈ 实现 = 低杠杆
```

**优先设计深模块**：一个深模块胜过 N 个浅模块。

### 4. 接缝纪律

> 一个适配器 = 假设的接缝。两个适配器 = 真实的接缝。

不要为只有一个实现的地方创建"抽象"。只有当变化真实存在时才引入接缝。

### 5. 内部 vs 外部接缝

- **内部接缝**：模块内部，用于测试，不暴露到外部
- **外部接缝**：模块公共接口，允许扩展

不要把内部接缝暴露到外部接口。

### 6. 深度是接口的属性，不是实现的属性

深模块可以由小的、可 mock 的、可交换的内部组件构成——只是这些不是接口的一部分。

## 术语关系图

```
Module（模块）
    ├── Interface（接口）→ 1:1 关系
    │   └── Depth（深度）→ 衡量模块
    ├── Implementation（实现）
    ├── Seam（接缝）→ 接口所在位置
    │   └── Adapter（适配器）→ 满足接口的具体事物
    ├── Leverage（杠杆）→ 调用者获得
    └── Locality（局部性）→ 维护者获得
```

## 深度定义

> 深度 = leverage at the interface = 行为/接口复杂度比

**深模块**：调用者或测试可以透过小接口 exercising 大量的行为
**浅模块**：接口几乎和实现一样复杂，没有杠杆

**衡量方式**：删除测试
- 删除深模块 → 复杂性在调用者中重现（好）
- 删除浅模块 → 复杂性消失（坏，透传）

## 拒绝的表述

| 错误的表述 | 原因 | 正确的表述 |
|-----------|------|-----------|
| 深度 = 实现行数/接口行数 | 奖励填充实现 | 深度 = 杠杆/接口 |
| 接口 = TypeScript interface 或 public methods | 太窄 | 接口 = 调用者必须知道的一切 |
| Boundary | 与 DDD 的 bounded context 混淆 | Seam 或 Interface |

## 架构设计检查点

### 模块划分

1. **理解这个功能需要看几个模块？**（越少越好）
2. **有没有只是透传的模块？**（合并到调用者）
3. **模块间是否有不必要的耦合？**
4. **模块命名是否使用领域术语？**

### 接口设计

1. **接口是否足够小？**（3-5 个方法以内）
2. **调用者需要记住多少？**（越少越好）
3. **有没有把实现细节泄露到接口上？**
4. **接口约束是否明确？**

### 深度评估

1. **删除这个模块会怎样？**（集中复杂性还是分散？）
2. **模块是"深"还是"浅"？**
3. **测试能覆盖多少行为？**

### 接缝位置

1. **哪里可能变化？**（需要可替换）
2. **这个接缝是否值得？**（至少两个适配器）
3. **内部细节是否隐藏好了？**
