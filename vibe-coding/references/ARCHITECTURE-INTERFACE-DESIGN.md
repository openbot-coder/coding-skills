# 接口设计方法

基于 "Design It Twice" (Ousterhout)——第一个想法不太可能是最好的。

## 核心思想

重要接口设计两次，选取最优方案。对比时关注：
- **深度**：谁的接口更小，杠杆更高？
- **局部性**：谁的变更集中在一处？
- **接缝位置**：谁更好地隐藏了变化？

## 流程

### 1. 定义问题空间

在并行生成设计前，先写用户-facing 的问题空间说明：

- 新接口需要满足的约束
- 依赖项及分类（见 ARCHITECTURE-DEEPENING.md）
- 粗略的代码草图（不是提案，只是让约束具体化）

展示给用户，然后立即进入 Step 2。用户阅读思考时，并行生成设计。

### 2. 并行生成设计

生成 3+ 个设计，使用不同约束：

#### Agent 1: 最小接口

> "最小化接口——最多 1-3 个入口点。最大化每个入口点的杠杆。"

**目标**：极简接口，深度优先

#### Agent 2: 最大灵活性

> "最大化灵活性——支持多种用例和扩展。"

**目标**：适应多种场景

#### Agent 3: 优化常见用例

> "为最常见的调用者优化——让默认情况变得 trivial。"

**目标**：简化常见路径

#### Agent 4: Ports & Adapters（如适用）

> "围绕跨接缝依赖设计 ports & adapters。"

**目标**：支持多种实现

### 3. 设计输出格式

每个设计输出：

```markdown
## 设计 N: [名称]

### 接口
- 类型、参数、返回值
- 不变量
- 错误模式
- 调用顺序（如有要求）

### 使用示例
```代码
// 调用者如何使用
```

### 隐藏了什么
- 接缝后面是什么
- 实现细节如何隐藏

### 依赖策略
- 依赖项及分类
- Adapter 列表

### 权衡
| 方面 | 评价 | 说明 |
|------|------|------|
| 深度 | 高/中/低 | 为什么 |
| 局部性 | 好/中/差 | 为什么 |
| 灵活性 | 高/中/低 | 扩展点在哪 |
```

### 4. 呈现与比较

顺序呈现设计，让用户消化每个，然后用散文对比：

| 维度 | 设计1 | 设计2 | 设计3 |
|------|--------|--------|--------|
| 深度 | | | |
| 局部性 | | | |
| 接缝位置 | | | |

### 5. 推荐

给出明确推荐和理由。如不同设计的元素组合良好，提出混合方案。

## 设计约束检查

### 必须满足

- [ ] 接口是否完整？（调用者需要的功能都有）
- [ ] 接口是否一致？（命名、错误处理风格统一）
- [ ] 接口是否可测试？（能否只通过接口验证行为）

### 应该满足

- [ ] 接口是否足够小？（3-5 个方法以内）
- [ ] 接口是否稳定？（不太可能频繁变更）
- [ ] 调用者需要记住多少？（越少越好）

### 最好满足

- [ ] 接口是否自文档化？（方法名清晰）
- [ ] 是否有良好的错误消息？（失败时给足够信息）

## 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 贫血接口 | 接口太大，什么都不做 | 合并到调用者或深化 |
| 泄漏抽象 | 调用者需要知道内部实现 | 隐藏细节 |
| 过度参数化 | 接口暴露不必要的灵活性 | 按需暴露 |
| 不一致命名 | 不同方法用不同风格 | 统一命名模式 |
| 忽略错误 | 不定义错误模式 | 明确可能的失败 |

## 接口评估表

| 评估项 | 1-2 | 3-4 | 5 |
|--------|-----|-----|---|
| 方法数量 | 太多 | 适中 | 太少（可能缺失功能） |
| 参数数量 | 太多 | 适中 | - |
| 深度 | 浅 | 深 | - |
| 自文档化 | 差 | 好 | - |
