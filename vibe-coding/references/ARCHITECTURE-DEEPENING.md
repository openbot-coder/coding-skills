# 模块深化方法

如何安全地将浅层模块集群深化为深层模块。

## 依赖分类

深化前，先评估模块的依赖类型。依赖类型决定测试策略。

### 1. 进程内依赖 (In-process)

**特征**：纯计算、内存状态、无 I/O

**深化策略**：始终可深化——合并模块，通过新接口直接测试

**测试**：无需 adapter，直接通过接口测试

### 2. 本地可替代依赖 (Local-substitutable)

**特征**：有本地测试替身（PGLite 替代 Postgres、内存文件系统）

**深化策略**：可深化，测试使用替身运行在测试套件中

**测试**：内部接缝（测试替身），无外部端口

### 3. 远程但自有的依赖 (Ports & Adapters)

**特征**：自有服务间的网络边界（微服务、内部 API）

**深化策略**：在接缝处定义 port，深模块拥有逻辑，传输作为 adapter 注入

**建议**：
> "在接缝处定义 port，实现生产 HTTP adapter 和测试内存 adapter，逻辑放在一个深模块中，即使部署跨越网络边界。"

**测试**：内存 adapter + 生产 HTTP adapter

### 4. 真正的外部依赖 (Mock)

**特征**：第三方服务（Stripe、Twilio 等），无法控制

**深化策略**：深模块将外部依赖作为注入的 port，测试提供 mock adapter

**测试**：mock adapter

## 接缝纪律

### 核心原则

> **一个 adapter = 假设的接缝。两个 adapter = 真实的接缝。**

不要引入接缝，除非有实际的东西跨它变化。

### 内部 vs 外部接缝

```
深模块
├── 外部接缝（公共接口）
│   └── 多个 adapter 实现
└── 内部接缝（私有实现）
    └── 测试替身
```

- 深模块可以有**内部接缝**（测试用）和**外部接缝**（扩展用）
- 不要把内部接缝暴露到外部接口
- 内部接缝只在模块内部使用

## 测试策略：替换，而非分层

### 旧测试的处理

浅层模块的旧单元测试在深模块的新接口测试存在后会成为冗余——删除它们。

### 新测试的位置

在深模块的接口层级编写新测试。**接口即测试表面。**

### 测试原则

1. **测试通过接口** — 调用者和测试跨同一接缝
2. **断言可观测结果** — 通过接口验证行为，而非内部状态
3. **测试应存活于重构** — 描述行为，而非实现

## 深化操作指南

### Step 1: 识别浅层模块集群

```
症状：
- N 个小模块，每个只有 2-3 个方法
- 调用者在多个模块间跳转才能理解一个概念
- 透传函数分散复杂性而非集中
```

### Step 2: 评估依赖

```
问：这个模块集群依赖什么类型的资源？
- 进程内？→ 直接合并
- 本地可替代？→ 检查替身是否存在
- 远程自有？→ 需要 ports & adapters
- 外部？→ 需要 mock 策略
```

### Step 3: 确定接缝位置

```
原则：
- 只在变化真实存在的地方创建接缝
- 接缝数量最少化
- 深模块内部不要随意拆分
```

### Step 4: 设计新接口

```
问：
- 调用者需要什么？
- 接口应该暴露多少？
- 如何隐藏实现细节？
```

### Step 5: 实现与迁移

```
1. 创建新深模块
2. 将逻辑迁移到新模块
3. 更新调用者使用新接口
4. 编写新测试在接口层级
5. 删除旧单元测试
6. 删除旧浅层模块
```

## 深化决策表

| 依赖类型 | 是否可深化 | 测试策略 | 接缝类型 |
|----------|-----------|----------|----------|
| 进程内 | ✅ 始终 | 直接测试接口 | 无需接缝 |
| 本地可替代 | ✅ 有替身时 | 替身测试 | 内部 |
| 远程自有 | ✅ 始终 | 内存 adapter | 外部 |
| 外部第三方 | ✅ 始终 | mock adapter | 外部 |
