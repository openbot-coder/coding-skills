# 任务拆解（Task Breakdown）

## 概述

**目的**：将设计文档拆解为可执行的任务清单，为阶段3（代码执行）做准备。

**重要**：这是一个专门的任务拆解技能，不是通用的规划技能。它专注于将已批准的设计转化为具体的任务列表。

<HARD-GATE>
在任务拆解之前，必须先完成设计文档（review-design 阶段）。任务拆解是对已有设计的细化，不是重新规划。
</HARD-GATE>

## 任务清单模板

所有任务拆解必须记录在 `{name}-progress.md` 的任务清单部分：

```markdown
### 任务清单

| 序号 | 任务名称 | 功能点数 | 依赖任务 | 状态 |
|------|----------|----------|----------|------|
| 1 | 任务名称 | 15 | - | ⏳ |
| 2 | 任务名称 | 12 | 任务1 | ⏳ |
```

### 任务详情模板

```markdown
#### 任务{序号}：{任务名称}

**功能点数**：{数量}

**功能列表**：
- [ ] 功能点1
- [ ] 功能点2
- [ ] 功能点3

**依赖**：
- {前置任务编号或 "无"}

**验证方式**：
- 测试覆盖：正例、反例、边界值
```

## 检查清单

任务拆解必须完成以下每一项：

1. **确认设计文档已批准** — 检查 `{name}-design.md` 是否存在且已通过 review-design
2. **提取功能要点** — 从设计文档中提取核心功能
3. **拆分任务** — 将功能分配到独立的任务（每个任务 10~20 个功能点）
4. **定义依赖关系** — 明确任务之间的依赖
5. **为每个任务添加功能列表** — 列出具体的功能点
6. **定义验证方式** — 为每个功能点指定测试覆盖类型

## 流程

### 1. 确认设计文档

```
检查以下文件是否存在且完整：
- {name}-design.md（设计文档）
- {name}-progress.md（进度文件，阶段2部分已初始化）
```

### 2. 提取功能要点

从设计文档中提取核心功能点：

```markdown
## 设计功能要点

从 `{name}-design.md` 中提取：
- 目标
- 核心功能
- 用户交互
- 数据流
- 边界条件
```

### 3. 拆分任务

**任务拆分原则**：

| 原则 | 说明 |
|------|------|
| 独立性 | 每个任务可独立验证 |
| 粒度 | 每个任务 10~20 个功能点 |
| 可测试 | 每个功能点有明确的验证方式 |
| 依赖明确 | 任务间依赖清晰标注 |

**避免与 Agent plan 冲突**：

- **这个技能**：只拆解已批准的设计，不做方案选择
- **Agent plan**：用于通用规划、问题分析
- **拆分任务**：精确到功能点列表，不是模糊的里程碑

### 4. 定义验证方式

为每个功能点指定测试覆盖类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| 正例 | 正常输入 → 正常输出 | `add(1, 2) → 3` |
| 反例 | 无效输入 → 错误响应 | `add("a", 1) → TypeError` |
| 边界值 | 边界条件测试 | `add(0, 0)`, `add(MAX, 1)` |

## 与 vibe-coding 的衔接

```
writing-design → review-design → task-breakdown → execute → verify → archive
```

**task-breakdown 完成后的操作**：

1. 运行 `python scripts/plans.py --name <变更名称>` 确认任务清单
2. 检查 `{name}-progress.md` 中的任务清单是否完整
3. 确认无误后，进入阶段3：`python scripts/execute.py --name <变更名称> --action list`

## 关键原则

- **专注拆分**：只拆解已有设计，不做方案选择
- **精确到点**：每个功能点都要列出
- **避免模糊**：不使用模糊的里程碑，使用具体的功能点
- **验证先行**：每个功能点提前想好验证方式
- **依赖清晰**：明确标注任务间的依赖关系

## 常见错误

| 错误 | 正确做法 |
|------|----------|
| 把方案设计当成任务拆解 | 设计已完成，拆解只细化实现 |
| 任务粒度过大（>20个功能点） | 拆分成多个独立任务 |
| 任务粒度过小（<10个功能点） | 合并相关功能到同一任务 |
| 依赖关系不明确 | 明确标注依赖的任务编号 |
| 功能点无验证方式 | 每个功能点指定测试类型 |
