# 审查设计

## 概述

设计审查是对 `{name}-design.md` 进行独立评审，确保设计内容完整、清晰、可执行。

**审查必须基于调研记录和领域术语，确保编写标准和审查标准一致。**

> **参考文档**：
> - [架构审查清单](../references/ARCHITECTURE-CHECKLIST.md) — 模块设计、接口设计、深度评估、接缝设计检查项
> - [模块深化方法](../references/ARCHITECTURE-DEEPENING.md) — 如何处理深化候选
> - [调研记录格式](../references/SURVEY-RECORDS.md) — 调研记录模板
> - [领域术语格式](../references/CONTEXT-FORMAT.md) — 术语定义格式

**重要原则：编写与评审必须由不同的 Agent 执行。**

## 审查前准备

### 必须读取的文件

1. **`{name}-survey-records.md`** — 调研记录
   - 原始对话记录
   - 术语澄清记录
   - 场景测试记录
   - 决策记录

2. **`CONTEXT.md`** — 领域术语（如存在）
   - 项目定义的术语
   - 避免使用的同义词

3. **`{name}-design.md`** — 设计文档

### 审查一致性检查

| 调研记录内容 | 设计文档应有 |
|-------------|-------------|
| 术语澄清 | 使用澄清后的术语，未使用避免词 |
| 场景测试 | 场景被覆盖或有理由排除 |
| 决策记录 | 设计体现该决策 |
| 待确认事项 | 状态已更新或说明原因 |

## 审查迭代流程

审查是一个**编写Agent与Reviewer Agent之间反复修改文档直到通过**的过程：

```
编写Agent → review-design Agent → 问题 → 修改 → 重新审查
                                              ↓
                                         通过审查
                                              ↓
                                       请求用户批准
                                              ↓
                                    进入阶段2：任务拆解
```

**用户批准是进入下一阶段的唯一入口。Agent 审查通过后，必须等待用户明确批准，才能进入任务拆解阶段。**

## 审查内容

### 1. 调研覆盖检查

检查调研记录与设计的一致性：

- [ ] **术语一致性**：设计是否使用 CONTEXT.md 中定义的术语？
- [ ] **术语避免**：是否避免了 CONTEXT.md 中的"避免词"？
- [ ] **场景覆盖**：调研中的场景是否在设计中被覆盖？
- [ ] **决策体现**：调研中的决策是否在设计中体现？
- [ ] **待确认处理**：调研中的待确认事项是否有明确结论？

### 2. 设计完整性检查

| 检查项 | 要求 | 权重 |
|--------|------|------|
| **目标** | 清晰描述变更要达成什么，1-3 句话 | 必须 |
| **背景** | 说明为什么需要这个变更 | 必须 |
| **成功标准** | 包含可验证的验收条件 | 必须 |
| **范围-包含** | 明确列出变更会做什么 | 必须 |
| **范围-不包含** | 明确边界，不做什么 | 推荐 |
| **影响范围** | 是否影响已有功能，用户是否知情 | 必须 |
| **非功能需求** | 性能、安全、兼容性等（如适用） | 可选 |

### 3. 设计质量检查

| 检查项 | 问题类型 | 严重程度 |
|--------|----------|----------|
| 目标模糊 | "优化用户体验" 无法验证 | 高 |
| 成功标准不可测 | "系统运行正常" 无具体指标 | 高 |
| 范围不明确 | 包含和不包含缺失 | 高 |
| 假设未声明 | 依赖外部系统但未说明 | 中 |
| 风险未识别 | 可能的失败场景未提及 | 中 |
| 安全风险 | 引入新的攻击面或漏洞 | 高 |
| 影响范围 | 影响已有功能但未告知用户 | 高 |

### 4. 术语使用检查

> 读取 `CONTEXT.md` 进行检查

- [ ] **术语正确性**：是否使用项目定义的术语？
- [ ] **避免词检查**：是否避免了 `_Avoid_` 中列出的词？
- [ ] **新术语处理**：如需新增术语，是否已记录？

### 5. 架构质量检查

> 详见 [架构审查清单](../references/ARCHITECTURE-CHECKLIST.md)

#### 模块设计
- [ ] 模块划分是否合理？有无过度/不足拆分？
- [ ] 模块命名是否使用领域术语？
- [ ] 有无浅层模块（只是透传）？

#### 接口设计
- [ ] 接口是否清晰、可测试？
- [ ] 接口是否足够小（3-5 个方法以内）？
- [ ] 接口约束是否明确？

#### 深度评估
- [ ] 删除测试：删除模块会集中还是分散复杂性？
- [ ] 杠杆分布：哪些深，哪些浅？
- [ ] 测试覆盖潜力如何？

#### 接缝设计
- [ ] 可扩展点是否预留接缝？
- [ ] 接缝位置是否合理（至少两个适配器）？
- [ ] 内部 vs 外部接缝是否区分正确？

#### 一致性
- [ ] 与现有架构风格一致？
- [ ] 与调研决策一致？

## 审查问题清单

### 基础问题

1. **目标清晰吗？** — 能否用一句话复述？
2. **背景充分吗？** — 为什么现在要做？
3. **成功标准可验证吗？** — 能通过测试验证？
4. **范围明确吗？** — 包含和不包含清晰？
5. **假设合理吗？** — 基于事实而非推测？
6. **风险可控吗？** — 有缓解措施？
7. **有安全风险吗？** — 有无新攻击面？
8. **影响范围已确认吗？** — 用户是否知情同意？

### 术语问题（基于调研记录）

9. **术语与调研一致吗？**
   - 设计中的术语是否与 survey-records 中的澄清一致？
   - 是否使用了 CONTEXT.md 中的标准术语？

10. **场景覆盖完整吗？**
    - survey-records 中的场景测试是否被设计覆盖？
    - 被排除的场景是否有合理解释？

11. **决策是否体现？**
    - survey-records 中的决策是否在设计中体现？
    - 与决策不一致的地方是否有理由？

### 架构问题

12. **模块划分合理吗？** — 理解功能需要看几个模块？
13. **接口设计够小吗？** — 调用者需要记住多少？
14. **深度是否平衡？** — 删除模块会怎样？
15. **接缝位置是否合理？** — 是否值得抽象？

## 审查规则

1. **编写者不能评审自己的文档** — 必须由不同的 Agent 执行
2. **审查发现问题时，由编写Agent修改文档**
3. **修改完成后重新提交给审查Agent继续审查**
4. **反复修改-审查循环，直到审查Agent通过**
5. **审查通过后，必须请求用户批准才能进入下一阶段**
6. **用户批准是进入任务拆解阶段的唯一入口**
7. **用户批准后必须通过 Git 提交进行版本管理**
8. **审查必须基于调研记录，确保编写标准和审查标准一致**

## 与 vibe-coding 的衔接

**审查通过后，更新设计状态表：**

当 Agent 审查通过时：
1. 将"Agent 审查"行的 ⏳ 改为 ✅
2. 填写审查通过的日期

当用户批准时：
1. 将"用户批准"行的 ⏳ 改为 ✅
2. 填写批准日期
3. **执行 Git 提交**：
```bash
git add {name}-design.md {name}-survey-records.md
git commit -m "feat: {name} 设计已获批准"
```

**审查完成后，进入任务拆解阶段：**
```
writing-design → review-design → 任务拆解 → 代码执行 → 测试验证 → 需求归档
```

**任务拆解阶段操作：**
1. 运行 `python scripts/plans.py --name <变更名称>` 创建计划
2. 基于通过审查的设计制定实施计划
3. 补充架构方案、实施步骤、依赖、风险

## 关键原则

- **审查必须基于调研** — 以 survey-records 和 CONTEXT.md 为准
- **术语一致性** — 设计与调研使用相同术语
- **审查是必需的** — 设计需要审查通过后才能进入计划阶段
- **编写与审查分离** — 由不同的 Agent 执行
- **反复修改是正常的** — 好设计是审查-修改迭代出来的
- **用户批准是最终** — Agent 审查通过后，请求用户批准才进入下一阶段
