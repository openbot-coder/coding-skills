# 审查设计

## 概述

设计审查是对 `{name}-design.md` 进行独立评审，确保设计内容完整、清晰、可执行。

**重要原则：编写与评审必须由不同的 Agent 执行。**

## 审查迭代流程（关键）

审查是一个**编写Agent与Reviewer Agent之间反复修改文档直到通过**的过程：

```plaintext
┌─────────────────────────────────────────────────────────────┐
│ 1. 设计编写 Agent 编写设计文档草稿                          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. review-design Agent 评审                                │
└─────────────────────────────────────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
              ▼                               ▼
        ⚠️ 需修改                        ✅ 通过
              │                               │
              ▼                               │
    ┌─────────────────┐                       │
    │ 3. 返回编写Agent │                       │
    │    修改文档      │                       │
    └─────────────────┘                       │
              │                               │
              └───────────────┬───────────────┘
                              ▼
              ┌─────────────────────────────┐
              │ 4. 重新提交给 review-design Agent │
              │    继续评审                  │
              └─────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
              ▼                               ▼
        ⚠️ 需修改                        ✅ 通过
              │                               │
              └───────────────┘               │
              （继续修改-评审循环）           │
                                              ▼
                              ┌─────────────────────────────┐
                              │ 5. 请求用户批准              │
                              │    "设计已通过审查，请确认" │
                              └─────────────────────────────┘
                                              │
              ┌───────────────────────────────┴───────────────────────────────┐
              │                                                               │
              ▼                                                               ▼
        ⚠️ 需修改                                                        ✅ 批准通过
        返回修改                                                           │
              │                                                             ▼
              │                            ┌─────────────────────────────┐
              │                            │      进入阶段2：任务拆解    │
              │                            └─────────────────────────────┘
              └───────────────┐
              （继续修改-评审-批准循环）
```

**关键：用户批准是进入下一阶段的唯一入口。Agent 审查通过后，必须等待用户明确批准，才能进入任务拆解阶段。**

## 审查内容

### 设计完整性检查

| 检查项 | 要求 | 权重 |
|--------|------|------|
| **目标** | 清晰描述变更要达成什么，1-3 句话 | 必须 |
| **背景** | 说明为什么需要这个变更，当前问题或机会 | 必须 |
| **成功标准** | 包含可验证的验收条件，可测试 | 必须 |
| **范围-包含** | 明确列出变更会做什么 | 必须 |
| **范围-不包含** | 明确边界，不做什么 | 推荐 |
| **影响范围** | 是否影响已有功能，用户是否知情同意 | 必须 |
| **非功能需求** | 性能、安全、兼容性等要求（如适用） | 可选 |

### 设计质量检查

| 检查项 | 问题类型 | 严重程度 |
|--------|----------|----------|
| 目标模糊 | "优化用户体验" 无法验证 | 高 |
| 成功标准不可测 | "系统运行正常" 无具体指标 | 高 |
| 范围不明确 | 包含和不包含缺失 | 高 |
| 假设未声明 | 依赖外部系统但未说明 | 中 |
| 风险未识别 | 可能的失败场景未提及 | 中 |
| 假设检验 | 验证假设是否合理 | 中 |
| 安全风险 | 引入新的攻击面或漏洞 | 高 |
| 影响范围 | 影响已有功能但未告知用户 | 高 |

### 审查问题清单

1. **目标清晰吗？**
   - 能否用一句话复述这个变更的目标？
   - 目标是否可量化或可验证？

2. **背景充分吗？**
   - 是否说明了为什么现在要做这个变更？
   - 是否有当前问题的具体描述或机会的论证？

3. **成功标准可验证吗？**
   - 每条标准是否有明确的验收条件？
   - 能否通过测试/检查来验证通过与否？

4. **范围明确吗？**
   - 是否清楚列出包含和不包含的内容？
   - 边界是否清晰，不会引起歧义？

5. **假设合理吗？**
   - 是否声明了关键假设？
   - 假设是否基于事实而非推测？

6. **风险可控吗？**
   - 是否识别了主要风险？
   - 是否有缓解措施或备选方案？

7. **有安全风险吗？**
   - 变更会引起关联的安全风险吗？
   - 是否有潜在的攻击面增加？
   - 是否有数据泄露、权限提升等风险？

8. **影响范围已确认吗？**
   - 是否影响以前的功能？
   - 用户调研时，是否已了解到变更影响的范围？
   - 用户是否明确同意接受变更影响范围？

```markdown
## 审查通过条件

设计必须满足以下所有条件才能通过审查：
- [ ] 所有"必须修复"问题已修复
- [ ] 目标清晰、可验证
- [ ] 成功标准可通过测试验证
- [ ] 范围边界明确
- [ ] 关键假设已声明
```

## 审查规则

1. **编写者不能评审自己的文档** — 必须由不同的 Agent 执行
2. **审查发现问题时，由编写Agent修改文档**
3. **修改完成后重新提交给审查Agent继续审查**
4. **反复修改-审查循环，直到审查Agent通过**
5. **审查通过后，必须请求用户批准才能进入下一阶段**
6. **用户批准是进入任务拆解阶段的唯一入口**
7. **用户批准后必须通过 Git 提交进行版本管理**

## 与 vibe-coding 的衔接

**审查通过后，必须更新设计状态表：**

当 Agent 审查通过时，在 `{name}-design.md` 的"设计状态"表中：
1. 将"Agent 审查"行的 ✅ 改为 ✅（之前是 ⏳）
2. 填写审查通过的日期

当用户批准时：
1. 将"用户批准"行的 ⏳ 改为 ✅
2. 填写批准日期
3. **执行 Git 提交**：将设计变更提交到版本管理

```bash
cd docs/vibe-coding/changes/{name}/{name}-design.md
git add {name}-design.md
git commit -m "feat: {name} 设计已获批准"
```

**审查完成后，进入任务拆解阶段：**

```
writing-design → 需求分析 → review-design → 任务拆解 → 代码执行 → 测试验证 → 需求归档
```

**任务拆解阶段操作：**
1. 运行 `python scripts/plans.py --name <变更名称>` 创建计划
2. 基于通过审查的设计制定实施计划
3. 补充架构方案、实施步骤、依赖、风险

## 关键原则

- **审查是必需的** — 设计需要审查通过后才能进入计划阶段
- **编写与审查分离** — 由不同的 Agent 执行，编写者不能审查自己的文档
- **反复修改是正常的** — 好设计是审查-修改迭代出来的
- **用户批准是最终** — Agent 审查通过后，请求用户批准才进入下一阶段
